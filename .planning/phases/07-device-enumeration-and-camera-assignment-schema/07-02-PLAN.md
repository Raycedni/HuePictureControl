       ```python
       class CameraDevice(BaseModel):
           device_path: str
           stable_id: str
           display_name: str
           connected: bool
           last_seen_at: str | None

       class CamerasResponse(BaseModel):
           devices: list[CameraDevice]
           identity_mode: str  # "stable" | "degraded"

       class ReconnectRequest(BaseModel):
           stable_id: str

       class ReconnectResponse(BaseModel):
           connected: bool
           device_path: str | None
           display_name: str

       class AssignmentRequest(BaseModel):
           camera_stable_id: str
           camera_name: str

       class AssignmentResponse(BaseModel):
           entertainment_config_id: str
           camera_stable_id: str
           camera_name: str
       ```

    3. Implement `GET /api/cameras` (DEVC-02, DEVC-03):
       - Wrap `enumerate_capture_devices()` in `await asyncio.get_event_loop().run_in_executor(None, enumerate_capture_devices)` to avoid blocking the event loop
       - For each V4L2DeviceInfo, call `get_stable_id(info.device_path, info.bus_info, info.card)` — also in executor since it may read sysfs files
       - Track `any_degraded = True` if any device returns `sysfs_available=False`
       - Upsert each device into `known_cameras` using `INSERT INTO known_cameras ... ON CONFLICT(stable_id) DO UPDATE SET display_name=excluded.display_name, last_seen_at=excluded.last_seen_at, last_device_path=excluded.last_device_path` with `datetime.now(timezone.utc).isoformat()` for last_seen_at. Per D-09.
       - Fetch all rows from `known_cameras` to include previously-seen-but-disconnected devices
       - Mark each device `connected: True` if its stable_id appears in current scan results, `connected: False` otherwise. Per D-06.
       - Set `identity_mode` to `"degraded"` if any_degraded or if no devices found and sysfs dir `/sys/class/video4linux/` does not exist, else `"stable"`
       - Return `CamerasResponse`

    4. Implement `POST /api/cameras/reconnect` (DEVC-05, D-04, D-05):
       - Accept `ReconnectRequest` body with `stable_id`
       - Re-run full scan (same as GET logic — call enumerate_capture_devices + get_stable_id)
       - Find device matching `stable_id` in scan results
       - If found: upsert known_cameras with new device_path and last_seen_at, return `ReconnectResponse(connected=True, device_path=..., display_name=...)`
       - If not found: look up display_name from known_cameras table (preserved per D-06), return `ReconnectResponse(connected=False, device_path=None, display_name=...)`
       - If stable_id not in known_cameras either: return 404

    5. Implement `PUT /api/cameras/assignments/{entertainment_config_id}` (CAMA-02):
       - Accept `AssignmentRequest` body
       - Validate that `camera_stable_id` exists in `known_cameras` (return 404 if not)
       - Upsert into `camera_assignments`: `INSERT INTO camera_assignments ... ON CONFLICT(entertainment_config_id) DO UPDATE SET camera_stable_id=excluded.camera_stable_id, camera_name=excluded.camera_name`
       - Return 200 with `AssignmentResponse`

    6. Implement `GET /api/cameras/assignments/{entertainment_config_id}` (CAMA-03):
       - Query `camera_assignments` for the given config_id
       - If row exists: return `AssignmentResponse`
       - If no row: return 404 with `{"detail": "No camera assignment for this config. Default capture device will be used."}` — this documents the CAMA-03 fallback contract
