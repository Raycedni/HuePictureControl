# Phase 03.1: Auto-Mapping from Entertainment Config - Research

**Researched:** 2026-03-24
**Domain:** Hue CLIP v2 channel positions, coordinate mapping, FastAPI REST, SQLite persistence, minimal React UI
**Confidence:** HIGH (core algorithm and data model), MEDIUM (coordinate ranges via third-party evidence)

---

## Summary

Phase 03.1 inserts a self-contained auto-mapping layer between the working Phase 3 streaming engine and the Phase 4 canvas editor. The goal is to produce usable `regions` and `light_assignments` rows in SQLite — the exact same tables that `StreamingService._load_channel_map()` already reads — by deriving polygon coordinates from channel position data in the Hue entertainment configuration, without any manual drawing.

The Hue CLIP v2 `/clip/v2/resource/entertainment_configuration` endpoint returns a `channels` array. Each channel carries a `channel_id` integer and a `position` object with three floats: `x`, `y`, `z`. The coordinate space is a 3D room model: `x` is left/right in [-1, 1], `y` is vertical (floor=−1, ceiling=+1), and `z` is depth (viewer=−1, screen=+1). For a "screen" configuration type the Hue app places lights at `y ≈ 1` (TV height) and spreads them across `x` and `z`. The mapping algorithm normalises `x` → screen column and `z` → screen row, then generates a rectangular polygon centred on that point.

The critical backend gap is that `list_entertainment_configs()` currently discards the `channels` array. A new function `fetch_entertainment_config_channels()` must fetch the full resource and return `[{channel_id, position: {x, y, z}}]`. The rest of the phase is persistence, a trigger endpoint, and a minimal preview page.

**Primary recommendation:** Use the Hue `x` coordinate (left/right, -1..+1) to map to screen x-axis, and the Hue `z` coordinate (depth, -1..+1) to map to screen y-axis. Generate square sampling polygons of fixed relative size (e.g. 20% of the screen) centred on each mapped point. Store results into existing `regions` and `light_assignments` tables via `INSERT OR REPLACE`.

### Pattern 1: Hue Coordinate → Normalized Screen Coordinate

**What:** Map the Hue 3D position to a 2D normalized screen coordinate [0..1].

**Coordinate system facts (MEDIUM confidence — verified from HyperHDR discussion logs and aiohue data model):**
- `x` axis: left = −1, right = +1. Map to screen x: `screen_x = (x + 1.0) / 2.0`
- `z` axis: depth front (viewer) = −1, back (screen) = +1. For lights placed behind the viewer to illuminate the screen, the Hue app places them in a spread across `z`. In practice for TV entertainment configs `z` spans the horizontal width of the setup; treat it identically to `x` when `x` is near zero, or swap axes. The safest interpretation: use `x` directly for left/right spread when lights are behind the viewer. **If `x` range is small (< 0.1 spread), fall back to `z`.**
- `y` axis: ignored — vertical position is irrelevant for screen-region mapping.

**Example (from HyperHDR API log):**
```json
{
  "channel_id": 0,
  "position": { "x": -0.21547, "y": 1.0, "z": -1.0 }
}
```
Maps to: `screen_x = (-0.21547 + 1.0) / 2.0 = 0.392`, `screen_y` derived from `z`.

**When to use:** Always for SCREEN-type entertainment configs. Apply identical logic to OTHER/3DSPACE configs.

**Example:**
```python
# Source: derived from Hue CLIP v2 coordinate space documentation
# x in [-1,1] -> [0,1], z in [-1,1] -> [0,1] (y ignored)
def channel_pos_to_screen(x: float, z: float) -> tuple[float, float]:
    screen_x = (x + 1.0) / 2.0
    screen_y = (z + 1.0) / 2.0
    return screen_x, screen_y
```

### Pattern 2: Screen Point → Sampling Polygon

**What:** Create a square polygon centered on a screen point with a configurable half-size.

**When to use:** For each channel after coordinate mapping.

**Example:**
```python
# Source: project convention from color_math.build_polygon_mask docstring
# Polygon points as [[x, y], ...] normalized [0..1], clamped to frame bounds
def make_square_polygon(cx: float, cy: float, half: float = 0.10) -> list[list[float]]:
    """Generate a square sampling region centred on (cx, cy).

    Args:
        cx: Normalized screen x [0..1]
        cy: Normalized screen y [0..1]
        half: Half-size of square as fraction of screen (default 0.10 = 10%)

    Returns:
        4-point polygon [[x0,y0],[x1,y0],[x1,y1],[x0,y1]]
    """
    x0 = max(0.0, cx - half)
    y0 = max(0.0, cy - half)
    x1 = min(1.0, cx + half)
    y1 = min(1.0, cy + half)
    return [[x0, y0], [x1, y0], [x1, y1], [x0, y1]]
```

### Pattern 3: Fetch Full Entertainment Config from Bridge

**What:** Extend `hue_client.py` with a function that returns raw channel data including `position`.

**When to use:** Called once by the auto-mapping trigger endpoint.

```python
# Source: Hue CLIP v2 API — existing pattern in hue_client.py
async def fetch_entertainment_config_channels(
    bridge_ip: str, username: str, config_id: str
) -> list[dict]:
    """Fetch channel positions for a specific entertainment configuration.

    Returns:
        List of dicts: [{channel_id: int, position: {x: float, y: float, z: float}}]
    """
    url = f"https://{bridge_ip}/clip/v2/resource/entertainment_configuration/{config_id}"
    headers = {"hue-application-key": username}
    async with httpx.AsyncClient(verify=False, timeout=10) as client:
        response = await client.get(url, headers=headers)
        data = response.json()
    item = data["data"][0]
    return [
        {
            "channel_id": ch["channel_id"],
            "position": ch["position"],
        }
        for ch in item.get("channels", [])
    ]
```

### Pattern 4: Persist Regions and Assignments

**What:** Write auto-generated regions and assignments into the existing SQLite tables. Use `INSERT OR REPLACE` (by region_id derived from config+channel) to make the operation idempotent.

**Key constraint (REGN-04):** Polygon must be stored as JSON of normalized [0..1] points.

```python
# Source: existing database.py schema + streaming_service.py json.loads pattern
import uuid, json

async def persist_channel_regions(
    db,
    config_id: str,
    channels: list[dict],  # [{channel_id, position: {x, y, z}}]
    half: float = 0.10,
) -> int:
    """Write one region + assignment row per channel. Returns count written."""
    written = 0
    for ch in channels:
        cid = ch["channel_id"]
        pos = ch["position"]
        screen_x, screen_y = channel_pos_to_screen(pos["x"], pos["z"])
        polygon = make_square_polygon(screen_x, screen_y, half)
        # Deterministic region_id: config_id:channel_id
        region_id = f"auto:{config_id}:{cid}"
        await db.execute(
            "INSERT OR REPLACE INTO regions (id, name, polygon, order_index) VALUES (?,?,?,?)",
            (region_id, f"Channel {cid}", json.dumps(polygon), cid),
        )
        await db.execute(
            """INSERT OR REPLACE INTO light_assignments
               (region_id, channel_id, entertainment_config_id) VALUES (?,?,?)""",
            (region_id, cid, config_id),
        )
        written += 1
    await db.commit()
    return written
```

### Anti-Patterns to Avoid
- **Using `y` coordinate for screen mapping:** `y` represents vertical room height and is near-constant (~1.0) in screen configurations. Using `y` as a screen axis produces incorrect clustering. Use `x` and `z` only.
- **Fetching channel positions from the SQLite cache:** The `entertainment_configs` table stores `raw_json`, but it was inserted by `list_entertainment_configs()` which discards `channels` data. Either re-fetch from bridge or fix the cache insertion. Re-fetching per trigger is simpler and correct for Phase 3.1.
- **Deleting existing regions on re-run:** Use `INSERT OR REPLACE` with deterministic `region_id` values (e.g. `auto:{config_id}:{channel_id}`) so re-running the endpoint is safe and idempotent without destroying manually-created regions from Phase 4.

## Common Pitfalls

### Pitfall 1: Wrong Axis for Screen Mapping
**What goes wrong:** Using `x`/`y` instead of `x`/`z` produces all regions clustered at the same vertical position (y ≈ 1.0 is constant for TV configurations).
**Why it happens:** The Hue coordinate `y` is room height (floor=−1, ceiling=+1), not screen vertical position.
**How to avoid:** Always use `x` for left/right and `z` for depth/front-back. For a screen-facing setup, `z` variation encodes horizontal positioning when `x` variation is small.
**Warning signs:** All auto-generated polygons overlap at the same y-position.

### Pitfall 2: Channel Positions Vary by Entertainment Config Type
**What goes wrong:** A "music" config places lights evenly in a circle; a "3dspace" config uses arbitrary 3D positioning. Neither maps cleanly to a screen rectangle.
**Why it happens:** The Hue app creates configs for different use cases with different spatial semantics.
**How to avoid:** Log a warning if `configuration_type` is not `"screen"`. Still proceed with the same axis mapping — it will be "good enough" and the user can fix via the Phase 4 editor.
**Warning signs:** Regions visually spread in unexpected directions on the preview page.

### Pitfall 3: Empty Channel List
**What goes wrong:** Calling auto-map on a config with 0 channels returns success but writes nothing.
**Why it happens:** Entertainment configs can exist without assigned lights.
**How to avoid:** Return HTTP 422 with detail message if `channels` is empty.

### Pitfall 4: Coordinate Range Assumptions
**What goes wrong:** If all lights are in a small cluster, dividing by the range to normalize over-spreads them; if range is zero (single light), division by zero occurs.
**Why it happens:** Users may create entertainment configs with just one or two lights.
**How to avoid:** The simple linear mapping `(x + 1.0) / 2.0` is bounded and never divides by zero. Avoid "stretch to fill" normalization that divides by the observed range.

### Pitfall 5: Re-fetch vs Cached Raw JSON
**What goes wrong:** The `entertainment_configs` SQLite table has a `raw_json` column that appears to store the full response — but inspection of `list_entertainment_configs()` in `hue_client.py` shows it only stores a summary dict (id, name, status, channel_count), not the full CLIP v2 JSON.
**Why it happens:** The `raw_json` column was created in the schema for future use but is not populated with channel data.
**How to avoid:** Always fetch channel positions directly from the bridge in the auto-map endpoint. Do not rely on the cached `raw_json`.

### Pitfall 6: Concurrent Start + Auto-Map
**What goes wrong:** If streaming is active when auto-map is called, `StreamingService` has already loaded its `channel_map`. The new regions won't be picked up until streaming is restarted.
**Why it happens:** `_load_channel_map` runs once at stream start.
**How to avoid:** The auto-map endpoint should return a warning if `streaming.state != "idle"` telling the user to stop/restart streaming to pick up new regions. Do not silently succeed.

## Code Examples

### Full Auto-Mapping Service Function

```python
# Source: derived from existing hue_client.py and color_math.py patterns
import json
import logging
import httpx

logger = logging.getLogger(__name__)

async def auto_map_entertainment_config(
    db,
    bridge_ip: str,
    username: str,
    config_id: str,
    polygon_half: float = 0.10,
) -> int:
    """Fetch channel positions and write regions + assignments to SQLite.

    Args:
        db: aiosqlite connection (from app.state.db)
        bridge_ip: Bridge IP address
        username: Hue application key
        config_id: UUID of the entertainment configuration
        polygon_half: Half-size of generated polygons in normalized [0..1] space

    Returns:
        Number of regions written.

    Raises:
        ValueError: If the config has no channels.
        httpx.HTTPStatusError: On bridge communication failure.
    """
    channels = await fetch_entertainment_config_channels(bridge_ip, username, config_id)

    if not channels:
        raise ValueError(f"Entertainment config {config_id} has no channels")

    written = 0
    for ch in channels:
        cid = ch["channel_id"]
        pos = ch["position"]
        screen_x = (pos["x"] + 1.0) / 2.0
        screen_y = (pos["z"] + 1.0) / 2.0

        # Clamp to [0,1]
        screen_x = max(0.0, min(1.0, screen_x))
        screen_y = max(0.0, min(1.0, screen_y))

        # Generate square polygon
        x0 = max(0.0, screen_x - polygon_half)
        y0 = max(0.0, screen_y - polygon_half)
        x1 = min(1.0, screen_x + polygon_half)
        y1 = min(1.0, screen_y + polygon_half)
        polygon = [[x0, y0], [x1, y0], [x1, y1], [x0, y1]]

        region_id = f"auto:{config_id}:{cid}"
        await db.execute(
            "INSERT OR REPLACE INTO regions (id, name, polygon, order_index) VALUES (?,?,?,?)",
            (region_id, f"Channel {cid}", json.dumps(polygon), cid),
        )
        await db.execute(
            """INSERT OR REPLACE INTO light_assignments
               (region_id, channel_id, entertainment_config_id) VALUES (?,?,?)""",
            (region_id, cid, config_id),
        )
        written += 1

    await db.commit()
    logger.info("Auto-mapped %d channels for config %s", written, config_id)
    return written
```

## Open Questions

1. **Axis swap: when does `z` vs `x` drive left/right?**
   - What we know: In HyperHDR discussion logs, lights behind a TV show `z` varying while `x` varies less. In a typical TV setup lights are placed at the left, right, and behind the TV.
   - What's unclear: Whether the Hue app always uses `x` for left/right or swaps to `z` depending on config type or room orientation.
   - Recommendation: Map both `x` and `z` using the formula `(val + 1) / 2`. Use `x` for screen_x and `z` for screen_y. Log the raw positions in the auto-map response so the user can verify visually via the preview page. Phase 4 allows manual adjustment if incorrect.

2. **`raw_json` column — populate or leave empty?**
   - What we know: The column exists in `entertainment_configs` table with `NOT NULL` constraint, but the row is never actually inserted by current code (the `GET /api/hue/configs` endpoint reads live from bridge and never writes to this table).
   - What's unclear: Whether Phase 3.1 should populate the `entertainment_configs` table as a side effect of auto-map.
   - Recommendation: Do not write to `entertainment_configs` in Phase 3.1. The auto-map function reads from the bridge directly. Populating this cache is a separate concern.

3. **Polygon half-size default**
   - What we know: 10% half-size = 20% width square. For a 5-channel config this gives moderate overlap.
   - What's unclear: Whether users will find overlap acceptable or distracting.
   - Recommendation: Hard-code 0.10 for Phase 3.1. Expose as a query parameter to the trigger endpoint so the user can experiment. Phase 4 allows manual resize.
