<objective>
Close gap **G-13-01** (VERIFICATION.md): `V4L2Capture` hardcodes MJPEG 640x480 via `VIDIOC_S_FMT` and decodes only MJPEG with `cv2.imdecode`. scrcpy `--v4l2-sink` writes raw YUV at the phone's native resolution to v4l2loopback, and because the kernel module is loaded with `exclusive-caps=1` the consumer's `S_FMT` returns `OSError: [Errno 22] Invalid argument`. Even if `S_FMT` were skipped, the reader cannot decode raw YUV with `cv2.imdecode`. This blocks SC-3 (Hue lights driven from the Android screen), blocks human UAT items 2 and 3, and will re-break Phase 14 (Miracast) which uses the same v4l2loopback consumer path.

Purpose: Make `V4L2Capture` format-agnostic so it works with both physical UVC cameras (MJPEG) and v4l2loopback devices whose format is owned by the producer (YUYV from many sources, YU12/I420 from scrcpy). Preserve the existing MJPEG path exactly — no regression for `/dev/video0`.

Output: A single-file change to `Backend/services/capture_v4l2.py` plus a new `Backend/tests/test_capture_v4l2.py` that covers the gap fix and the physical-camera regression. After this plan, a real POST to `/api/wireless/scrcpy` with a running scrcpy producer no longer throws EINVAL, the virtual device becomes readable, and the existing `StreamingService -> CaptureRegistry -> V4L2Capture -> Hue` pipeline delivers frames to the lights.

Reference: VERIFICATION.md section `### Gaps Summary` -> `#### G-13-01 -- V4L2Capture cannot consume scrcpy's raw YUV output (BLOCKER)` (Proposed fix, Option A).

<pixel_format_reference>
Every constant below is load-bearing. Use these exact values in the source code.

| Format     | FourCC bytes (little-endian uint32) | Hex literal    | OpenCV decode                                 |
|------------|-------------------------------------|----------------|-----------------------------------------------|
| MJPEG      | 'MJPG' = 4D 4A 50 47 (LE)           | 0x47504A4D     | `cv2.imdecode(arr, cv2.IMREAD_COLOR)`         |
| YUYV       | 'YUYV' = 59 55 59 56 (LE)           | 0x56595559     | `cv2.cvtColor(arr, cv2.COLOR_YUV2BGR_YUYV)`   |
| YU12 (I420)| 'YU12' = 59 55 31 32 (LE)           | 0x32315559     | `cv2.cvtColor(arr, cv2.COLOR_YUV2BGR_I420)`   |

Buffer sizes (raw YUV, no headers):
- YUYV: 2 bytes per pixel -> `width * height * 2` bytes; reshape to `(H, W, 2)` before `cvtColor`.
- YU12 / I420: planar 4:2:0, 1.5 bytes per pixel -> `width * height * 3 // 2` bytes; reshape to
  `(H * 3 // 2, W)` before `cvtColor(..., COLOR_YUV2BGR_I420)`.

Errno:
- `errno.EINVAL == 22`. Use `errno.EINVAL` (import `errno`), not the magic number.
</pixel_format_reference>

<ioctl_reference>
VIDIOC_G_FMT has magic 'V' (0x56), nr=4, direction IOR|IOW, struct size 204 bytes for the
v4l2_format struct plus type prefix (the S_FMT / G_FMT ioctls share the same struct layout).
The existing code writes a 208-byte buffer and reads back width/height/pixelformat from
offsets 4, 8, and 12 respectively. Use the same buffer size and offsets for G_FMT.

Numerical value: `_VIDIOC_G_FMT = 0xC0D05604` (same as `_VIDIOC_S_FMT` with nr byte 0x05 -> 0x04).
Alternative equivalent: compute with `_iowr(ord("V"), 4, 204)` if the executor prefers to
reuse the existing `_iowr` helper — but note the helper in this file uses `size << 16` with
`_v4l2_buf_size` (sizeof v4l2_buffer, 88 bytes), not the 204-byte v4l2_format size. Prefer
the hard-coded literal `0xC0D05604` to match the style of `_VIDIOC_S_FMT = 0xC0D05605`.
</ioctl_reference>

```python
# Try to negotiate MJPEG 640x480 (physical UVC cameras). If the producer
# owns the format (v4l2loopback with --exclusive-caps=1, as scrcpy uses),
# the kernel returns EINVAL -- fall back to G_FMT to adopt the producer's
# pixel format, width, and height. See VERIFICATION.md G-13-01.
fmt = bytearray(208)
struct.pack_into("<I",  fmt, 0,  _V4L2_BUF_TYPE_VIDEO_CAPTURE)
struct.pack_into("<II", fmt, 4,  _WIDTH, _HEIGHT)
struct.pack_into("<I",  fmt, 12, _V4L2_PIX_FMT_MJPEG)

try:
    fcntl.ioctl(fd, _VIDIOC_S_FMT, fmt)
    # S_FMT succeeded -- the kernel may still have rounded w/h/fmt, so read back.
    self._width       = struct.unpack_from("<I", fmt, 4)[0] or _WIDTH
    self._height      = struct.unpack_from("<I", fmt, 8)[0] or _HEIGHT
    self._pixelformat = struct.unpack_from("<I", fmt, 12)[0] or _V4L2_PIX_FMT_MJPEG
    logger.info(
        "S_FMT negotiated pixelformat=0x%08X %dx%d",
        self._pixelformat, self._width, self._height,
    )
except OSError as exc:
    if exc.errno != errno.EINVAL:
        raise
    # Producer owns the format (e.g. v4l2loopback exclusive-caps=1 fed by scrcpy).
    # Read the current format via G_FMT.
    gfmt = bytearray(208)
    struct.pack_into("<I", gfmt, 0, _V4L2_BUF_TYPE_VIDEO_CAPTURE)
    fcntl.ioctl(fd, _VIDIOC_G_FMT, gfmt)
    self._width       = struct.unpack_from("<I", gfmt, 4)[0]
    self._height      = struct.unpack_from("<I", gfmt, 8)[0]
    self._pixelformat = struct.unpack_from("<I", gfmt, 12)[0]
    logger.info(
        "S_FMT rejected (EINVAL) -- producer owns format; G_FMT reports "
        "pixelformat=0x%08X %dx%d",
        self._pixelformat, self._width, self._height,
    )

if self._pixelformat not in (_V4L2_PIX_FMT_MJPEG, _V4L2_PIX_FMT_YUYV, _V4L2_PIX_FMT_YU12):
    raise RuntimeError(
        f"Unsupported V4L2 pixel format 0x{self._pixelformat:08X} "
        f"on {self._device_path}; supported: MJPEG, YUYV, YU12 (I420)"
    )
```
