### Fallback: Explicit Devices List
Per D-02 and D-06, document when to use and how to switch:

**When to switch:** If `docker compose exec backend ls /dev/video*` shows "No such file" despite hardware being attached (cgroup rules grant permission but some Docker environments don't auto-mount device nodes).

**How to switch:** Replace `device_cgroup_rules` block in docker-compose.yaml with:
```yaml
    devices:
      - "/dev/video0:/dev/video0"
      - "/dev/video2:/dev/video2"   # Add one line per capture card
```
Note: Explicit devices require container restart when adding new cards. Each device path must match the host path.

## WSL2 / usbipd Walkthrough
Per D-05, include step-by-step commands with example output:

### Attaching USB Devices to WSL2
```powershell
# PowerShell (Administrator recommended):
usbipd list
```
Example output:
```
Connected:
  BUSID  VID:PID    DEVICE                          STATE
  1-2    1b3f:2008  USB Video, USB Audio             Not shared
  1-5    0bda:8153  Realtek USB GbE                  Not shared
```

```powershell
usbipd bind --busid 1-2        # One-time: enable sharing
usbipd attach --wsl --busid 1-2  # Attach to WSL2
```

### Verifying in WSL2
```bash
ls /dev/video*    # Should show /dev/video0, /dev/video1, etc.
```

### Verifying Inside Docker Container
```bash
docker compose exec backend ls /dev/video*
```

### Adding a Second Capture Card
Same process with the second device's BUSID:
```powershell
usbipd bind --busid 2-1
usbipd attach --wsl --busid 2-1
```

## Common Gotchas

### Device Path Shifts on Re-attach
When unplugging and re-attaching a capture card (or after `usbipd detach` + `attach`), the device path may change (e.g., `/dev/video0` becomes `/dev/video2`). This is normal Linux V4L2 behavior. Use the camera selector dropdown in the web UI to select the new device path. With the cgroup rules approach, no Docker restart is needed.

### WSL Restart Drops USB Attachments
After `wsl --shutdown` or a Windows restart, all usbipd attachments are dropped. Re-run `usbipd attach --wsl --busid <busid>` for each capture card.

### Docker Desktop vs. Native Docker Engine
If using Docker Desktop (not Docker Engine installed directly in WSL2), USB devices attached to WSL2 may not be visible inside Docker containers because Docker Desktop runs in a separate lightweight VM. Verify with `docker compose exec backend ls /dev/video*`. If devices don't appear, ensure your version of usbipd-win supports Docker Desktop integration (usbipd-win 4.x+).

### Permission Denied on /dev/video*
If you see "Permission denied" when the backend tries to open a capture device, verify that `group_add: [video]` is present in docker-compose.yaml. The `video` group (GID 44 on standard Linux) provides filesystem-level access to V4L2 device nodes.
